关于使用软件的一点小想法

2014-12-14

要码字，使用什么字编辑器好呢，我这些日子一直用开了 Sublime Text ，作为普遍的文本处理器。可是用着用着感觉挺不爽的，它既没有 Vim 的那种酸爽流畅的感觉，也加多了一些莫名其妙的多数时候用不到的功能。它唯一的我发现的优点就是内置了很多编程语言与配置文本的语法高亮，这个在 Vim 上没有内置，可是需要的话可以自己去自定义或者下载插件。也许就是因为这个小小的功能让我禁不住诱惑使用了它，可是现在我彻底醒悟了。一样东西能流行好几十年是有它的道理的，而且还能在程序员的世界里引发无数的口水战。很多时候人就是这样，看不透事物的本质，一下子追那个潮流，一下子有新东西 published 就又跑去跟那个时尚，却始终注意不到真正的美，只存在静静的观摩中。或者说，人性上的缺点使得我们很容易因为一点点蝇头小利而上当受骗，看不住自己真正值钱的东西。Vim 编辑器就是这样的一款美得忽视了时间的产品，虽然很多人抱怨学习 Vim 的时间成本比较好，但是一旦熟练掌握，就能在字体里遨游穿梭而不受限制了。当然是除了有非常好的 IDE 外，一般的文本编辑工作在 Vim 的高手眼里简直就是小菜一碟。

跑题了，拉拉，回来啦。可是就算我曾经对 Vim 的各种模式和命令烂熟于心，还把《practical Vim》里面的全部技巧都制作成思维导图，当我要重新拾起它时，还是感觉对自己不满：为什么这个你忘了，那个你也忘了？哎，现在反省起来，这种对自己的责备真的毫不利己，人随着长大，慢慢的自尊大起来了，很多事情尝试一下，做不到就放弃了，并说自己不喜欢这样做。我就时常看到一些文章提到为什么孩子学语言的时候都是不断的说，不怕被指出错误还要继续说，在不断练习的帮助下，绝大部分孩子都能在三岁？左右熟练日常的话语。可是反思我们这些小大人呢？我从初中就开始学英语，至今起码有十年的英语学习时间，可是你要我跟一个以英语为第一语言的人聊天，我肯定顺溜不起来。巨大差距啊！

就像一个味的寻找什么书最好，到处看评论，不断的收集PDF，可是最终刚开始学习的热情就被这种寻找的过程给消耗了，结果什么都没学到。还像是不断的排斥现在要做的事情，认为自己真正喜欢的不是这个，整天想东想西，却不干真正的实事，思维也在这不断的彷徨无措，与挣扎中被弄得筋疲力尽。这样做还不如一开始就跟定一条路，好好研究，研究出个门道，掌握基本的方法论与思维方式，这些都是受用无穷的东西，可以在学习别的技术时产生迁移的元知识。要专注啊！要专注啊！刚刚写完第一个“要专注啊”就不知道神游去哪了，干。

似乎码字的时候没有注意逻辑，因为没有一个主心骨来写？不过我是带着想写关于 bug 的主题打开 Vim 的啊，总是絮叨最后可别搞得我自己都没有兴趣去回顾了。

我想说的其实是遇到问题，不要各种尝试，那个试一下，这个试一下，要不断的尝试说明你没有掌握到问题的核心本质，你不知道怎么才能解决它，盲目的尝试比摸着石头过河还要糟糕，毕竟河还是会有底的，可是软件系统问题多多，你怎么知道用哪种方式去解决呢？这时候很重要的一点，就是 Read the Fucking manual! 系统设计的人才知道问题在哪里，哪里的输入错了会输出什么结果，还有就是要认真看编译器或者解释器提供的 Warming，一定要把 Warming 当作是 Error 来看待，养成良好的习惯，这样就可以在下次编程的自动的排除了错误了。如果在本地找不到解决方案的时候，再去分发软件的作者那里寻求答案，如果有当然最好了，就算没有到 Stack Overflow 也会有大量的已知问题已经被解决的了。

其实我昨天遇到的博客问题就是，博文写完了，推送了，可页面就是不 Auto-generation，烦恼，这很麻烦，又调试不了。这里问题就来了，一开始我猜测是文本文件的编码问题（为什么要猜是这个问题呢？乱来），可在我尝试复制原本就存在的 blog 在提交后，还是没有显示新的文章。那就不是文本的问题啦？肯定是啦，我最常犯的一个错误就是不断的尝试同一样事情（明知结果也会一样，在电脑这个精确运行的系统上），不断的试啊试啊，现在想想真跟傻逼似的，傻逼，傻逼，砍手。

然后上去 github 看文档，看看 github pages 的说明，有没有什么东西在最近几个月改动了，弄得系统不合我心意。然后就看到一个提示，提醒大家自己下一个 jekyll 服务器下来运行一下，那样在 build 的过程中出现了什么问题也好自己改正过来。很好，这不就是改 bug 过程中的，bug 必须重现才可以进行修改嘛。记住”repeatability“。可是好麻烦好麻烦，为什么 rubyGem 下不来包呢？一直 stuck 在控制台那里，换成 powershell 也不行，换成管理员执行也不行。真揪心，可我没有放弃，继续探究。然后今天早上终于搞定下好各种组件了，就开始在本地运行服务器，打开浏览器进行本来 ip loop 浏览。问题依旧！！这可就恼人了。然后我再怀疑 Byte order Mark 的问题，果然，新增的文件竟然在 Utf-8 的编码前面多增加了三个字节。修改，去除三个字节，哇，太好了，出来了。可是 push 到远端，一看还是没有在 pages 服务器上 Generate 新的页面。这就太不顺利了，然后我就死盯着 jekyll 提示的错误，好像说 _config.yaml 文件里面的 Auto-generation 被禁用了。哦，这时候，最大的可能不就是因为配置文件就错了一点？ 干，还真是。删除一行，一切完美。

真的是流水账似的，不过在这过程中的不耐烦还无法用文字直接的表达出来，这些不耐烦都是非常不必要的，要记住啊。

为什么软件的用户界面的那么的不友好，说改配置就该配置呢，不是应该一直保存兼容性的嘛。而且文档上至少要标出来改动过程啊，标出哪年哪月改动了，然后用户就可以很方便的按图索骥的进行相应的修改。
